# ポートフォリオサイト レスポンシブ対応方針

## 目的

本サイトは、情報を並べるだけのページではなく、タイルを触る・展開する・移動させる体験そのものをデザインの中心にしている。

そのため、レスポンシブ対応ではPC版レイアウトを単純に縮小して全端末へ適用するのではなく、端末ごとにタイルの見せ方・配置・展開体験を調整する。

重視すること：

- タイルの硬質感・ツヤ・目地に埋め込まれた印象を維持する
- About / Works / Blog / GitHub の役割を端末が変わっても保つ
- タイルを押した時の気持ちよさ、展開時の体験を維持する
- 文字の読みやすさとタップのしやすさを損なわない
- 端末ごとに無理のない配置へ切り替える

## 基本方針

レスポンシブ対応は、以下の3つのレイアウトモードとして設計する。

```text
PC版
タブレット版
スマホ版
```

ただし、HTMLページを3つ別々に作るのではない。

共通で使うもの：

- HTML構造
- JavaScriptの基本処理
- About / Works / Blog / GitHub の役割
- タイルクリック・展開・閉じる・外部遷移の操作ルール

画面サイズごとに変更するもの：

- タイルの配置
- タイルサイズ
- 目地の幅
- 外側余白
- タイル角の大きさ
- サブタイルの数・配置
- 展開後タイルのサイズと位置
- 文字サイズ
- 詳細コンテンツの表示幅

実装は、共通HTML・共通JavaScriptを基本とし、CSSの `@media` によって3モードのレイアウトを管理する。

## やらないこと

- PC用HTML、タブレット用HTML、スマホ用HTMLを別々に作る
- 同じクリック処理を端末別に重複して書く
- PC版をそのまま極端に縮小してスマホ表示に使う
- タイル内文字が読めなくなるほど小さくする
- タップ領域が小さくなる配置を維持する
- レスポンシブ対応のために既存PC版を大きく崩す

## レイアウトモード

ブレークポイントの数値は、実装時に実機・DevTools確認をしながら調整する。

最初の区分候補：

```css
/* PC：基本レイアウト */
@media (min-width: 1024px) {
}

/* タブレット */
@media (min-width: 768px) and (max-width: 1023px) {
}

/* スマホ */
@media (max-width: 767px) {
}
```

この数値は固定仕様ではなく、タイルが読めるか・押しやすいか・展開表示が成立するかを見て調整する。

## PC版方針

### 位置づけ

現在作成中のレイアウトをPC版の基本完成形とする。

### 維持する体験

- 8列のタイルボード
- 多数のサブタイルを含むモザイク状配置
- 硬質なメインタイルと儚いサブタイルの対比
- About / Works 展開時に周囲のタイルが退避・再配置される演出
- Blog / GitHub の画面遷移アニメーション
- Works詳細を展開タイル内部でスクロール表示する構造

### PC版で優先すること

- 世界観の完成度
- タイル配置の美しさ
- アニメーションの気持ちよさ
- Works詳細の読みやすさ

## タブレット版方針

### 位置づけ

PC版の世界観をなるべく残しつつ、タップ操作や表示幅に合わせて整理したレイアウトにする。

### 検討方針

以下は実際に配置ラフを作って判断する。

- PC版の8列配置を維持できるか
- 6列程度に組み替える方が自然か
- サブタイルをすべて残すか、一部減らすか
- About / Works の展開サイズをどこまで大きくするか

### タブレット版で優先すること

- タップしやすいメインタイルサイズを保つ
- タイルのモザイク感を可能な範囲で残す
- 文字が小さくなりすぎない
- Works詳細画像カードが読める幅を確保する

### 想定する調整内容

- タイル1マスの大きさを縮小
- 目地幅を縮小
- 外側paddingを縮小
- タイル角を少し縮小
- 必要ならサブタイル配置を整理
- 展開タイルをPC版より画面幅寄りに広げる

## スマホ版方針

### 位置づけ

PC版の縮小表示ではなく、スマホで触りやすく読めるように組み替えた専用レイアウトにする。

### 基本方針

- 4列程度のタイル構成を基本候補にする
- About / Works / Blog / GitHub を読める大きさで配置する
- サブタイルは減らす、または装飾として限定表示する
- About / Works 展開時は画面幅の大部分を使う
- Works詳細は縦スクロールで読むことを優先する

### スマホ版で優先すること

- 押しやすさ
- 読みやすさ
- 展開後の閲覧性
- 同じ世界観を感じられること

### スマホ版で変えてよいこと

- タイルの並び順
- サブタイルの表示数
- サブタイルのサイズ
- 展開時に非対象タイルを見せ続けるかどうか
- 展開タイルの位置
- 外部遷移アニメーションの大きさ・速度

### スマホ版で維持すること

- About / Works / Blog / GitHub の役割
- タイルを押して反応する体験
- 展開中に他タイルがロック状態になるルール
- ロック中タイルの硬質な触感表現
- Blog / GitHub が外部遷移タイルであること

### スマホ版初期配置ラフ方針

スマホ版は、PC版の8列配置をそのまま縮小するのではなく、縦長のモザイク状タイルボードとして再構成する。

Figmaラフでは、配置した状態でタイルボードの上端と下端が画面内に見えることを重視する。スクロールしないと全体像が分からない状態ではなく、スマホでもWorks / Blog / GitHub / Aboutの存在が一覧として認識できる状態を目指す。

基本方針：

- 4列程度のグリッドを候補にする
- メイン4タイルは読める大きさを維持する
- サブタイルは世界観と目地感を残すために配置する
- ただし、一覧性を損なう場合はサブタイルを削る
- タイルボード全体が長くなりすぎる場合は、まず最上部のサブタイル列を削る

現在の配置ラフの考え方：

```text
上部：サブタイルとGitHub
中段：Blog
下段：WorksとAbout
```

Aboutは下側に配置する。スマホでは下側の方が親指で押しやすく、自己紹介は興味を持った後に触れる導線としても自然なためである。

ただし、Aboutの存在が分かりにくくならないように、画面内でタイルボード全体の上下が見えることを前提にする。

## CSS管理方針

既存のPC版CSSを基本スタイルとして扱い、タブレット版・スマホ版はファイル後半の `@media` 内で上書きする。

```css
/* =========================
   共通設定・PC基本レイアウト
========================= */

/* 現在の基本CSS */


/* =========================
   タブレットレイアウト
========================= */

@media (min-width: 768px) and (max-width: 1023px) {
  /* タブレット専用調整 */
}


/* =========================
   スマホレイアウト
========================= */

@media (max-width: 767px) {
  /* スマホ専用調整 */
}
```

### CSS class の扱い

- 通常の見た目・配置・構造は単独classで管理する
- レスポンシブ用の配置変更は `@media` 内で既存配置classを上書きする
- 状態変化は `is-*` class を維持する
- レスポンシブ対応のためだけにJavaScript側へ状態分岐を大量追加しない

## CSS変数の扱い

タイルの材質感やサイズ調整に関わる値は、各画面モードでCSS変数として管理する方向を検討する。

管理候補：

```css
:root {
  --tile-unit: 80px;
  --tile-gap: 12px;
  --tilespace-padding: 20px;
  --corner-cut: 10px;
}
```

タブレット・スマホでの上書きイメージ：

```css
@media (min-width: 768px) and (max-width: 1023px) {
  :root {
    --tile-unit: 64px;
    --tile-gap: 10px;
    --tilespace-padding: 16px;
    --corner-cut: 8px;
  }
}

@media (max-width: 767px) {
  :root {
    --tile-unit: 72px;
    --tile-gap: 8px;
    --tilespace-padding: 12px;
    --corner-cut: 7px;
  }
}
```

数値は仮置きであり、実際の配置モックと操作確認後に決定する。

## `clamp()` の利用方針

`clamp()` は、PCからスマホまで同一レイアウトを連続的に縮小するために使うのではなく、各画面モード内の微調整に使う。

使ってよい例：

```css
@media (max-width: 767px) {
  .tilespace {
    padding: clamp(8px, 3vw, 12px);
    gap: clamp(6px, 2vw, 8px);
  }

  .tile-title {
    font-size: clamp(12px, 3.5vw, 15px);
  }

  :root {
    --corner-cut: clamp(6px, 2vw, 8px);
  }
}
```

目的：

- 小さめスマホと大きめスマホの差を自然に吸収する
- タブレット内での幅差を微調整する
- タイル角や余白を不自然に固定しない

## タイル角のレスポンシブ方針

現在、タイルの角落としは `--corner-cut` を使用している。

タイルの角は、横長タイルだから大きく変えるのではなく、画面モードごとのタイルサイズに合わせて統一して調整する。

理由：

- タイル群は同じ材質・同じ加工のパーツとして見せたい
- 横長タイルだけ角を大きくすると統一感が崩れる
- スマホで全体が小さくなる場合のみ、角も少し小さくするのが自然

方針：

- PC：現在の角落としを基本とする
- タブレット：タイルサイズに合わせて少し小さくする
- スマホ：専用配置内でタイルの大きさに合う角へ調整する

## JavaScriptの扱い

JavaScriptの基本操作処理は端末ごとに分けない。

共通で維持する処理：

- タイル展開
- タイルを閉じる
- Worksページ切り替え
- Blog / GitHub 外部遷移
- `is-tile-locked` の付与・解除
- ロック中クリック抑止

PC版では、About / Works 展開時に周囲のタイルを退避・再配置している。スマホ版で配置を大きく変更した場合、同じ再配置先が成立しない可能性がある。

その場合は、いきなりJavaScript処理を大きく分岐させず、まず以下を検討する。

- CSSの配置変更だけで同じclass処理を使い回せないか
- スマホでは退避対象タイルを非表示にする方が自然ではないか
- 状態classは共通のまま、見た目だけ変えられないか

どうしても必要な場合のみ、端末別の動作差分を小さな責務で追加する。

## レスポンシブ実装へ入る前の作業順

1. PC版Works詳細ページの画像カード構成を完成させる
2. PC版の基本操作を確認する
   - 初期表示
   - About展開 / 閉じる
   - Works展開 / 一覧 / 詳細 / 戻る / 閉じる
   - Blog遷移
   - GitHub遷移
   - タイルロック中の触感と抑止処理
3. タブレット版のタイル配置ラフを作る
4. スマホ版のタイル配置ラフを作る
5. レイアウトラフに基づいてCSSのmedia queryを追加する
6. タブレット版を小さい差分で実装・確認する
7. スマホ版を小さい差分で実装・確認する
8. 最後に全モードの操作とアニメーションを確認する

## Codexへの作業指示

- PC版を基本レイアウトとして壊さない
- いきなり全画面分を実装しない
- タブレット版とスマホ版の配置ラフが決まってからCSSを書く
- HTMLを端末別に複製しない
- JavaScriptを端末別に大きく分岐させない
- media queryはタブレット用・スマホ用でまとまりを分ける
- 変更後は対象画面だけでなくPC版への影響も確認する
- 各モードでタイル展開・閉じる・ロック・外部遷移を確認する

## 現時点の結論

PC / タブレット / スマホを、3つの明確なレイアウトモードとして設計する。

ただし、HTMLと基本JavaScriptは共通で維持し、CSSのmedia queryで配置・サイズ・展開表現・余白を画面別に管理する。

PC版を完成させた後、タブレット版とスマホ版の配置ラフを作成してから段階的に実装へ進む。
