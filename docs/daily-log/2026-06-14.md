## 迷ったこと・苦労したこと

- タイル待避後の配置アニメーションが二重に見える原因の切り分けに時間がかかった。
  - 最初は `unlockTiles()` や `--return-delay` が原因に見えたが、実際には初期表示アニメーションの終了状態が残っていたことが原因だった。

- GitHub Pages上でPC幅では二重、スマホ幅では一重になる挙動から原因を絞り込んだ。
  - スマホ幅にすると `mobile-subtile1` が表示され、初期表示アニメーションの終了処理が発火していた。
  - PC幅では `mobile-subtile1` が非表示で、終了処理が発火していなかった。

- リンクボタンのデザイン調整で、リストマーカーやアイコン色の扱いを確認した。

## 判断したこと

- 初期表示アニメーションの終了判定は、最後のDOM要素の `animationend` に依存しない。
  - レスポンシブで表示・非表示が切り替わるタイルがあるため、時間で外す方が安定すると判断した。

- リンクボタンの見た目用クラスとJS用クラスは分けて管理する。
  - `work-link-button` は見た目用。
  - `work-github-link` はGitHub遷移アニメーションの対象を示すJS用。

## 学んだこと

- `display: none` の要素ではCSSアニメーションの `animationend` が発火しない場合がある。
  - レスポンシブで非表示になる要素を「最後のアニメーション完了判定」に使うと不具合につながる。

- 状態classが残り続けると、後続のアニメーションと重なって、別の不具合に見えることがある。
  - 今回は `is-intro-playing` が残ったことで、復帰時の `tile-assemble` と初期表示用の指定が重なっていた。

- DevTools Consoleで状態classの有無を確認すると、アニメーション不具合の切り分けがしやすい。

- リンクボタンのようなUIでは、見た目のclassと動作用classを分けると責務が分かりやすい。
