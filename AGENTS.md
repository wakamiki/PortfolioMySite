ユーザーは「完成コードを受け取ること」よりも、
「自分で考えて実装できるようになること」を重視している。
そのため、原則として以下を優先する。
- 考えるポイント
- タスク分解
- ヒント
- 詰まりやすい点

## コード出力方針
完成コードを一括で大量生成しない。
ただし、学習や実装確認に必要な小さな差分コード、短い関数、限定的なCSS修正は提示してよい。
コードを書く場合も以下を徹底する。
- 必要最小限にする
- 小さい単位で分ける
- 一度に大規模実装をしない
- 「なぜその書き方をするか」を説明する
- 学習目的を阻害しない
---
## HTML方針
HTMLは単なる見た目ではなく、
意味を持ったタグ構造として扱うこと。
HTMLはsemantic HTMLを意識する。
そのため以下を重視する。
- semantic HTML
- 適切なタグ選択
- divの乱用を避ける(ただし、装飾レイヤー、アニメーション用ラッパー、レイアウト補助など、意味を持たない要素には `div` / `span` を使ってよい。)
- アクセシビリティを意識する
- 構造と見た目を分離する
---
## CSS方針
CSSは「見た目を整える」だけでなく、
レイアウト・状態変化・UI体験を作るものとして扱う。
特に以下を重視する。
- CSS Grid
- Flexbox
- Transition
- Transform
- Keyframes
- レスポンシブ設計
- 状態変化によるUI設計

### CSS class 設計方針

CSS class は、できるだけ「1つの class につき1つの責務」を持たせる。

通常の見た目・配置・構造は、単独 class で管理する。
`.parent .child` のような複合セレクタは、状態変化・疑似状態・展開時の内部パーツ制御などに限定する。

避けたいこと：

- 単なる display 指定のために親子セレクタを使う
- 既存 class に別責務を追加して対応する
- 共通 class に特定画面だけの都合を混ぜる

推奨すること：

- 共通見た目は共通 class に書く
- 専用レイアウトは専用 class を作る
- 一時的な状態は `is-*` class にする
- 状態変化以外では、複合セレクタを増やしすぎない
---
## JavaScript方針
JavaScriptは「画面に動きと状態を与えるもの」として扱う。
- 状態 class を追加する時は、付けるタイミングだけでなく、外すタイミングも必ず決める。
- 特に `is-*` class は、アニメーション終了時、閉じる時、ページ切り替え時など、どの時点で解除するかを明確にする。
特に以下を重視する。
- 状態管理
- class切り替え
- イベント処理
- UIインタラクション
- DOM操作
- 小さな責務での関数分割
---
## JavaScript / CSS の役割分担方針
UIの不具合やアニメーション調整では、JavaScriptだけで解決しようとせず、CSS側で吸収できる問題かも必ず検討する。

JavaScriptは、イベント処理・状態 class の付け外し・必要なタイミング制御を担当する。
CSSは、見た目・レイアウト・transition / animation・opacity / transform / filter などの視覚変化を担当する。

修正時は以下を確認する。

- CSSの詳細度、animation-fill-mode、transition、opacity、transform で解決できないか
- JavaScriptで状態管理した方が分かりやすい問題か
- JSとCSSを組み合わせた方が自然なアニメーションではないか

原則として、JavaScriptは「状態を変える係」、CSSは「状態に応じた見た目を作る係」と考える。

UIの不具合調査では、JavaScriptの処理だけでなく、HTML構造、CSSの詳細度、display、position、z-index、animation-fill-mode、transition、transform の影響も確認する。

# 実装ルール
## 基本姿勢
大規模な自動変更は避ける。
ただし、原因調査、責務分離、重複削減、小さな差分実装は積極的に行う。
不明点がある場合は、作業を止めずに仮案・選択肢・確認ポイントを出す。
---
## 不要な変更を避ける
以下は禁止または極力避ける。
- 関係ないファイル変更
- 命名変更だけの修正
- 大規模一括変更
- 一度に複数責務を変更すること
---
## 小さい単位で変更する
変更は必ず段階的に行う。
推奨手順：
1. 小さく変更
2. 動作確認
3. 問題確認
4. 次の変更
一気に作り切らないこと。
---
## 変更内容を明示する
変更時は必ず以下を説明すること。
- 何を変更したか
- なぜ変更したか
- どこを確認すればよいか
- 次に何をするか
---
## 不明点の扱い
仕様・デザイン・挙動に曖昧さがある場合は、勝手に大きな変更をしない。
ただし、作業を止めずに進められる小さな仮実装や選択肢提示は行ってよい。
その場合は「仮置きであること」「確認すべき点」を明示する。
---
## リファクタリング方針

目的のないリファクタリングや、見た目・挙動に関係ない大規模整理は避ける。

ただし、重複処理の削減、責務分離、状態 class の整理、バグ予防につながる小さなリファクタリングは行ってよい。
その場合は、変更目的と確認方法を明示する。
---
### 既存処理の整理方針

関数化は、複数箇所で使う処理、または責務が明確な処理から行う。
1箇所でしか使わない処理を無理に共通化しない。

既存処理を関数化・共通化する時は、まず1箇所だけ置き換えて動作確認する。
見た目や挙動に問題がないことを確認してから、他の箇所へ展開する。
---
# UI / UX 方針
このプロジェクトでは、
「静かな見た目」と「触ると気持ちいい動き」を重視する。
特に以下を重視する。
- 小さな弾性
- 短いアニメーション
- 微小なオーバーシュート
- 心地よい停止感
- 統一された操作感
- シンプルな図形UI
- 余白を活かした設計
過度な演出や長すぎるアニメーションは避ける。
- UI変更後は、見た目に不具合がないことを確認してから次の変更へ進む。
- 特にアニメーション、展開、閉じる、ページ切り替えは、変更後に一通り操作確認する。
---
# アニメーション方針
アニメーションは「派手さ」ではなく、
「触り心地」を目的とする。
特に以下を重視する。
- Transition
- Transform
- Keyframes
- scaleによる弾性
- opacity変化
- 小さい減衰表現
揺れは必要最小限にすること。
---
# レスポンシブ方針
PC / タブレット / スマホで
「同じ見た目」ではなく、
「同じ操作感」を重視する。
共通化するもの：
- 操作ルール
- アニメーション方針
- UIの反応
変更してよいもの：
- 列数
- タイルサイズ
- 余白
- フォントサイズ
---
# 文字コード方針
文章ファイルは必ず UTF-8 で扱うこと。
特に PowerShell 環境では、
UTF-8 のファイルでも文字化けが発生する場合があるため注意すること。
以下を確認すること。
- UTF-8 保存
- BOM有無
- PowerShell の文字コード
- ターミナル表示文字コード
文字化けが発生した場合は、
ファイル内容ではなく読み込み側の問題も疑うこと。
