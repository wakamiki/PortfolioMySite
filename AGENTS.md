# AGENTS.md
## プロジェクト方針
このプロジェクトは、ポートフォリオサイトを通して
JavaScript / CSS / HTML の理解と自力実装力を高めることを目的とする。
特に以下を重視する。
- JavaScript の理解
- CSS の理解
- HTMLタグの適切な使い分け
- UI / UX の設計理解
- アニメーション実装の理解
- 自力で考えて実装する力
---
# 開発方針
## 学習優先
ユーザーは「完成コードを受け取ること」よりも、
「自分で考えて実装できるようになること」を重視している。
そのため、原則として以下を優先する。
- 考えるポイント
- タスク分解
- ヒント
- 詰まりやすい点
- デバッグ方法
- 確認方法
- 設計意図
---
## コード出力方針
完成コードを一括で大量生成しないこと。
コードを書く場合も以下を徹底する。
- 必要最小限にする
- 小さい単位で分ける
- 一度に大規模実装をしない
- 「なぜその書き方をするか」を説明する
- 学習目的を阻害しない
---
## HTML方針
HTMLは単なる見た目ではなく、
意味を持ったタグ構造として扱うこと。
そのため以下を重視する。
- semantic HTML
- 適切なタグ選択
- divの乱用を避ける
- アクセシビリティを意識する
- 構造と見た目を分離する
---
## CSS方針
CSSは「見た目を整える」だけでなく、
レイアウト・状態変化・UI体験を作るものとして扱う。
特に以下を重視する。
- CSS Grid
- Flexbox
- Transition
- Transform
- Keyframes
- レスポンシブ設計
- 状態変化によるUI設計
---
## JavaScript方針
JavaScriptは「画面に動きと状態を与えるもの」として扱う。
特に以下を重視する。
- 状態管理
- class切り替え
- イベント処理
- UIインタラクション
- DOM操作
- 小さな責務での関数分割
---
# 実装ルール
## 不要な変更を避ける
以下は禁止または極力避ける。
- 不要なリファクタリング
- 関係ないファイル変更
- 命名変更だけの修正
- 大規模一括変更
- 一度に複数責務を変更すること
---
## 小さい単位で変更する
変更は必ず段階的に行う。
推奨手順：
1. 小さく変更
2. 動作確認
3. 問題確認
4. 次の変更
一気に作り切らないこと。
---
## 変更内容を明示する
変更時は必ず以下を説明すること。
- 何を変更したか
- なぜ変更したか
- どこを確認すればよいか
- 次に何をするか
---
## 不明点は推測しない
仕様・デザイン・挙動に曖昧さがある場合、
勝手に決め打ちしないこと。
必要に応じて確認を行うこと。
---
# UI / UX 方針
このプロジェクトでは、
「静かな見た目」と「触ると気持ちいい動き」を重視する。
特に以下を重視する。
- 小さな弾性
- 短いアニメーション
- 微小なオーバーシュート
- 心地よい停止感
- 統一された操作感
- シンプルな図形UI
- 余白を活かした設計
過度な演出や長すぎるアニメーションは避ける。
---
# アニメーション方針
アニメーションは「派手さ」ではなく、
「触り心地」を目的とする。
特に以下を重視する。
- Transition
- Transform
- Keyframes
- scaleによる弾性
- opacity変化
- 小さい減衰表現
揺れは必要最小限にすること。
---
# レスポンシブ方針
PC / タブレット / スマホで
「同じ見た目」ではなく、
「同じ操作感」を重視する。
共通化するもの：
- 操作ルール
- アニメーション方針
- UIの反応
変更してよいもの：
- 列数
- タイルサイズ
- 余白
- フォントサイズ
---
# 文字コード方針
文章ファイルは必ず UTF-8 で扱うこと。
特に PowerShell 環境では、
UTF-8 のファイルでも文字化けが発生する場合があるため注意すること。
以下を確認すること。
- UTF-8 保存
- BOM有無
- PowerShell の文字コード
- ターミナル表示文字コード
文字化けが発生した場合は、
ファイル内容ではなく読み込み側の問題も疑うこと。
---
# 出力スタイル
優先する出力順：
1. 今何を作っているか
2. 作る機能
3. 触るファイル
4. 考えるポイント
5. タスク分解
6. ヒント
7. 詰まりやすい点
8. 確認方法
完成コードの大量出力は避けること。
